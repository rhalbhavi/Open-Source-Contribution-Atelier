## 🧲 Pull Requests — Sending Your Changes Home

A **Pull Request** (often shortened to PR) is how you say to a project: "Hey, I made some changes over in my fork, and I think they'd be great in your project too!"

Think of it like this: you borrowed a friend's recipe book and photocopied a page (that's your fork). You added your own twist to the recipe. Now you want to give them your improved version. A Pull Request is you handing that improved page back and saying, "Want to add this to your book?"

## 🎯 What Goes Into a Pull Request?

Every PR has a few key parts:

- **Title**: A clear, short summary of what you changed. Example: `feat: implement dark mode switcher`
- **Description**: Explains the *why* and *how*. Include links to related issues (e.g. `Closes #42` — this tells GitHub to automatically close issue #42 when the PR is merged).
- **Files Changed**: A side-by-side view showing exactly what lines you added, changed, or removed.
- **Reviewers**: Other contributors who will look at your code, ask questions, and either approve or request changes.

---

## 🛠️ How to Raise a Pull Request

1. **Push your branch** to your fork on GitHub and click **Compare & pull request**.
2. **Automated tests** (CI/CD) run in the background to check your code doesn't break anything.
3. **Reviewers** look at your code, leave comments, or ask for changes.
4. **You make updates** on your local branch and push them — the PR updates automatically.
5. A **maintainer clicks Merge**, and your code becomes part of the official project!

It's that simple. You fork, you change, you request — and the project grows better thanks to you.

---

### 🧠 Key Takeaway

A Pull Request is your formal proposal to add your changes to a project. It includes a title, description, the code changes themselves, and a space for reviewers to discuss. Every open source contribution starts with one.
