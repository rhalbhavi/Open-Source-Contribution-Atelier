## 📌 Issues — Where Contributions Begin

Before anyone writes a single line of code, most open source work starts in an **Issue**.

Think of an Issue like a **sticky note on a community bulletin board**. Someone posts: "The coffee machine is leaking" (a bug) or "It would be great if we had tea too" (a feature request). Everyone can see it, discuss it, and someone can volunteer to fix it.

---

## ❓ What is an Issue?

An **Issue** is a tracking ticket. It's used to:

- Report a bug
- Suggest a new feature
- Request documentation improvements
- Discuss bigger architectural decisions

It's the starting line for almost every contribution.

## 🎯 Parts of an Issue

Every issue has a few helpful parts:

- **Title**: A short name for the issue, set by whoever created it.
- **Issue Number**: A unique ID (like `#42`) that GitHub assigns automatically. You'll use this number to reference the issue in commits and PRs.

  <img width="648" height="74" alt="image" src="https://github.com/user-attachments/assets/b7dd959a-46a5-4692-a2e6-b4a599692063" />

- **🏷️ Labels**: Color-coded tags that categorize the issue — like `bug`, `enhancement`, `good first issue`, or `documentation`. These help you find the type of work you're looking for.

  <img width="400" height=auto alt="image" src="https://github.com/user-attachments/assets/07c5f4c0-6bdc-493b-82ed-cc5ffd7e9fe9" />

- **👤 Assignee**: The person who's currently working on this issue. If nobody is assigned, it's up for grabs!

- **📍 Milestones**: Target release goals that group related issues together — like "v2.0 launch."

## 🛠️ How to Raise an Issue

1. Go to the project's GitHub page and click the **Issues** tab.
2. Click the green **New Issue** button.
3. If the project offers templates (like "Bug Report" or "Feature Request"), pick the one that fits. If not, click **Open a blank issue**.
4. Write a clear title. Example: `Bug: Login button not working on Android phones`.
5. Fill in the description — explain what's happening, what you expected, and how to reproduce it.
6. Click **Submit new issue**.

---

> [!NOTE]
> Opening an issue is a contribution! If you find a bug and carefully explain how to reproduce it, you're saving maintainers valuable time. You don't need to write code to help.

---

### 🧠 Key Takeaway

Issues are the heartbeat of an open source project. They track bugs, ideas, and tasks. Before you code, look for an issue to work on — or open one yourself. That's where every great contribution begins.
