# Reading Guidelines First

Imagine you're invited to a friend's house for dinner. You wouldn't walk in and start rearranging their kitchen, right? You'd ask: "Where's the bathroom? What can I help with? Any house rules?"

Contributing to a project is the same. Before you write a single line of code, you need to read the project's rules. These live in two critical files.

---

### 📖 README.md — The Welcome Mat

The **README** (Read Me) is the first thing visitors see. It's like the front door of a house — it tells you what this place is all about.

A good README tells you:

- What the software does
- How to install and run it
- Quick usage examples

If the README is missing or unclear, that's a great opportunity for your first contribution — you could improve it!

---

### 📋 CONTRIBUTING.md — The Instruction Manual

The **CONTRIBUTING.md** file is the project's playbook for contributors. Think of it like the **assembly instructions** that come with new furniture. Ignore them, and you might end up with extra screws and a wobbly table.

This file explains:

- The project's code structure
- How to name your branches (e.g. `fix/...` instead of `bug/...`)
- How to run tests and check code style
- The Pull Request submission process

---

> [!WARNING]
> Submitting code that doesn't follow the project's style guide — or that breaks existing tests — wastes everyone's time. Always read `CONTRIBUTING.md` first. It's like checking the dress code before going to a party. A minute of reading can save hours of rework.

---

### 🧠 Key Takeaway

Before you start coding, read the README to understand what the project does, and the CONTRIBUTING.md to learn how they want contributions. A little preparation shows respect and sets you up for success.
